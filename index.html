Current challenges identified on discovery call
CHALLENGE 01
Logic locked in Qlik QVD generators
Business transformation logic is embedded in proprietary Qlik constructs. Axys must extract and re-express this as SQL — dbt becomes the new home for it.
CHALLENGE 02
Semantic models are 1:1 with reports
Each Qlik report has its own isolated model. The AI-readiness goal requires consolidating down to 8–10 shared semantic models with standardized naming and context.
CHALLENGE 03
No metadata or column context on legacy tables
Some older tables have generic, undescriptive column names. Without enriched metadata, AI systems like Claude and Copilot cannot reason accurately over the data.
CHALLENGE 04
Dual platform state for 2+ years
Redshift and Databricks will coexist for an extended period. The transformation layer needs to work across both without being tightly coupled to either.
CHALLENGE 05
No SLA on free dbt Core
Enterprise architect flagged this directly. For a production reporting migration with business-critical pipelines, community support carries meaningful operational risk.
CHALLENGE 06
Agentic development playbook still undefined
CDM Smith wants a 2027 playbook for AI-assisted BI development (using Claude + VS Code/Cursor, and Microsoft spec-driven agents). The data foundation layer is not yet locked in.
Page 02 — dbt Cloud Value

Where dbt Cloud fits into CDM Smith's architecture

dbt Cloud doesn't replace your BI layer — it strengthens the foundation beneath it, providing the metadata, governance, and context your AI initiatives depend on.

Transformation layer — before & after dbt Cloud
⚠ Qlik QVD Generators + ad-hoc SQL
  • Logic embedded in Qlik constructs — not portable, not testable
  • Per-report models with duplicated transformations and no reuse
  • No column definitions, lineage, or data quality testing
  • No metadata enrichment — AI and Copilot operate on blind tables
  • No SLA, no orchestration, no observability
  • Redshift and Databricks managed separately with no common layer
✓ dbt Cloud transformation layer
  • Modular, reusable SQL models with full lineage — source to column
  • Every model ships with a metadata YAML file: definitions, tests, column docs
  • Data quality tests run on every build — you know if data is healthy before it reaches Power BI or Claude
  • Agnostic to warehouse — runs identically on Redshift and Databricks during the dual-platform period
  • Semantic Layer serves metrics to Claude, Power BI, and Databricks Genie Room simultaneously
  • Enterprise SLA, native CI/CD, browser IDE, managed orchestration — no DIY infra

Page 03 — Core vs Cloud

dbt Core vs dbt Cloud — an honest comparison

dbt Core is real and capable. The question is whether the infrastructure you'd build around it is a better use of engineering time than the migration itself.

⚙️
dbt Core
The open-source SQL transformation engine. Powerful, free, and battle-tested. Runs your models, generates docs, tracks lineage. No scheduler, no IDE, no SLA, no Semantic Layer.
+
☁️
dbt Cloud
The full managed platform built on top of Core. Adds orchestration, browser IDE, CI/CD, enterprise SLA, Explorer observability, and the Semantic Layer. The engine plus the infrastructure around it.

RECOMMENDATION
Axys recommended Core because it works — and it does. But CDM Smith's enterprise scale, AI ambitions, and dual-platform complexity means you'll quickly outgrow the free tier. The question isn't whether Core can run your SQL. It's whether you want to spend the next two years maintaining the infrastructure around it, or investing that time in the migration and the AI use cases you described.
